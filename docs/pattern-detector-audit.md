# Pattern Detector Audit

Status: phase 1 baseline, active.

This document is the technical audit contract for Pattern Detector. It exists so detector changes are measured against repeatable cases instead of visual impression alone.

## Scope

Audited module:

- Pattern Detector backend scores: kick, tom, snare, hat, cymbal, global.
- Pattern Detector event behavior: first hit, repeated hits, tail retriggers, cross-class residues.
- Display Render dependencies that read Pattern Detector hits.

Out of scope for phase 1:

- Final threshold tuning.
- New UI controls.
- ML/NMF drum transcription.

## Professional Basis

The current detector is energy/flux based. That is not enough for tails and mixed percussion. The next detector must separate:

- Attack/onset: abrupt spectral change.
- Body/sustain: tonal or noisy energy after the attack.
- Tail/wash: continued energy that should not retrigger a new hit by itself.

Relevant references:

- Essentia `OnsetDetection`: HFC, complex-domain, flux, melflux, RMS methods for onset detection.
- Essentia onset tutorial: onset detection should be treated as an onset detection function plus peak picking.
- Librosa HPSS/percussive extraction: harmonic/percussive separation is a standard route for distinguishing sustained tonal material from percussive material.
- DAFx real-time drum transcription/NMF: template-based drum classification is a later option, not a phase 1 dependency.

## Metrics

The audit script must report:

- `predicted`: strongest detected class.
- `pass`: whether `predicted` matches expected class.
- `targetDominance`: expected class peak divided by strongest non-target peak.
- `peaks`: maximum displayed hit value per class.
- `hitCounts`: event count per class.
- `retriggersAfter250ms`: repeated event count after the initial attack window.
- `firstHit`: first event time per class.
- `tonalLowTailMax`: low-frequency sustained energy diagnostic.
- `highWashMax`: sustained high-frequency wash diagnostic.

## Pass/Fail Rules For Future Detector Work

Phase 1 is allowed to fail. It is a baseline.

After phase 2, a detector revision should not be accepted unless:

- Isolated kick predicts kick and tom residue is materially lower than kick.
- Isolated tom predicts tom and kick residue is materially lower than tom.
- Isolated snare predicts snare without hat/cymbal becoming co-dominant.
- Isolated hats predicts hat without cymbal becoming co-dominant.
- Isolated cymbal predicts cymbal, but repeated wash does not produce repeated onset hits.
- `synthetic_kick_plus_808_tail` has no repeated kick hits during the tail unless a new attack is present.
- `targetDominance` should generally be above `1.5` for isolated samples; exceptions require a written note.

## Architectural Finding

The current Pattern Detector calculates a `hit` from novelty of a class feature that still includes attack, body, and tail. This makes it vulnerable to:

- 808/sub tails retriggering kick.
- Clap/snare broadband attacks also lighting hat/cymbal.
- Kick/tom overlap in low and low-mid bands.
- Cymbal wash being treated as repeated cymbal hit instead of sustained presence.

The phase 2 backend must split event evidence from body evidence before class decisions.

## Phase 2 Baseline Result

Implemented in `src/app.js` and mirrored in `tools/pattern-audit.mjs`:

- Added class candidates before final features:
  - `kickCandidate`
  - `tomCandidate`
  - `snareCandidate`
  - `hatCandidate`
  - `cymbalCandidate`
- Added morphology gates:
  - snare body gate is shell/body based, not just high-frequency crack.
  - hat gate uses high-frequency ratio against snare body.
  - cymbal gate uses sustained air/wash evidence.
  - tom gate uses low-mid ring/body and suppresses sub-only kick tails.
- Added class competition:
  - snare body suppresses false cymbal/hat.
  - hat-like signals suppress false snare.
  - cymbal wash suppresses false hat.
  - snare crack/body suppresses false kick.

Current generated report:

```text

```

Known remaining issue:

- The raw class winner now passes the phase 1 fixtures, but `Peak Hit Values` still saturate to `1.000` across multiple classes. This means the visual/event envelope is still too permissive. Phase 3 should add class-aware hit competition and/or per-class event state machines so the displayed bars stop looking equally maxed when only one class is musically dominant.

## Phase 3 Event Competition Result

Implemented in `src/app.js` and mirrored in `tools/pattern-audit.mjs`:

- Added per-class competition profiles:
  - `close`: how close a non-winning class must be to remain active.
  - `floor`: minimum residual hit allowed for non-winning classes.
  - `decay`: residual decay multiplier for display hits.
- Added short temporal class lock:
  - kick: 160 ms
  - tom: 140 ms
  - snare: 180 ms
  - hat: 70 ms
  - cymbal: 220 ms
- A locked class suppresses other classes unless their feature clearly exceeds the locked feature.

Current generated report:

```text
Baseline classification failures: none
```

Improvements over phase 2:

- Isolated kick no longer maxes snare/hat/cymbal display hits.
- Isolated hats no longer maxes cymbal.
- Synthetic snare plus hats now keeps snare as feature, hit, and count winner.
- Synthetic kick plus 808 tail has no tail retriggers and no high-family display saturation.

Remaining limitation:

- Some samples still show brief first-frame or late-tail peak residuals, especially low-family kick/tom and high-family hat/cymbal. Removing those completely requires either a small decision latency/lookahead window or a fuller per-class attack/body/tail state machine. The current phase keeps latency at zero.

## Phase 4 Micro-Lookahead

Implemented in `src/app.js` and mirrored in `tools/pattern-audit.mjs`:

- Added a 12 ms decision queue for Pattern Detector events.
- Hits are no longer committed immediately.
- Candidate hits decay the display immediately, then enter `pendingHits`.
- When pending hits mature, the detector chooses the strongest short-window candidate and suppresses weaker competing classes.
- Per-class timing now enforces minimum event separation:
  - kick: 180 ms
  - tom: 160 ms
  - snare: 130 ms
  - hat: 45 ms
  - cymbal: 200 ms

This is not full DAFx/NMF drum transcription. The DAFx/NMF architecture would add learned or curated spectral templates for instruments and solve classification by template activation over time. This phase keeps MET-TR realtime and light, but moves the backend toward the same staged architecture:

```text
features -> class candidates -> short decision queue -> accepted events -> visual envelope
```

## Rejected Direction: Static Spectral Signature Layer

A fixed manual spectral signature layer was tested and rejected. It made the live detector worse by suppressing snare/hat/cymbal and reinforcing the wrong families in continuous music.

Do not reintroduce static hand-written templates as a replacement for the shared spectral backend. If MET-TR moves toward DAFx/NMF-style transcription, it should use measured reference activations from the local sample set or curated templates, with explicit CPU limits and an audit that includes continuous-music cases.

Current conservative fix after rollback:

- Tom sensitivity, gain, hold, and lock time are reduced.
- Tom now requires stronger low-mid ring/body and a better tom/sub ratio.
- Snare/hat/cymbal can escape kick/tom locks more easily, so the high family does not disappear during dense music.

## Phase 5 Continuous Pattern Audit

Implemented in `tools/pattern-audit.mjs`:

- `pattern-reference.mid` is parsed as event truth.
- `default-drums.wav` and `default-prog.wav` are decoded and analyzed as continuous incoming audio.
- The audit writes detector transcriptions:
  - `docs/generated/default-drums-detected.mid`
  - `docs/generated/default-prog-detected.mid`
- The report now includes global and per-class precision/recall against the MIDI truth.
- The report includes an alignment diagnostic that searches tempo/offset hypotheses without changing the official score.

Current result after adding family-level arbitration and corrected offline event timestamp:

```text
default-drums: 13/32 matched, precision 0.33, recall 0.41
default-prog:  12/32 matched, precision 0.35, recall 0.38
```

Current per-class reading:

- Kick is partially working: `default-drums` recall 0.56, `default-prog` recall 0.67.
- Tom is partially working but still confused with kick/body energy: recall 0.33 to 0.50.
- Snare is partially working: recall 0.38 in both continuous cases.
- Hat is not working in the continuous fixtures: recall 0.00.
- Cymbal is inconsistent: `default-drums` recall 1.00 with too many extras, `default-prog` recall 0.00.

This is still not acceptable. It is useful because it proves the isolated sample audit was too weak.

Architecture change mirrored in `src/app.js` and `tools/pattern-audit.mjs`:

- Removed the single global winner behavior from the event queue.
- Added class families:
  - Low: kick, tom.
  - Mid/high: snare, hat, cymbal.
- Competition and locks now suppress mainly inside their own family instead of muting the whole detector.
- This allows simultaneous events such as kick+snare, tom+cymbal, snare+hat to survive the decision queue.

Measured finding:

- 2048 FFT improves some continuous timing but breaks isolated tom identity.
- 4096 FFT improves some continuous matching but breaks too many isolated fixtures.
- 8192 FFT preserves class identity better but smears transient timing.

Conclusion:

- Pattern Detector needs a multi-resolution backend:
  - Short window for onset timing and transient confidence.
  - Longer window for class identity/body/tail.
  - A small state machine per class: attack, body, tail, cooldown.
- A single shared FFT window cannot satisfy both drum timing and class identity.
